## 💡 优劣势分析

### `g/` 分层架构优势

#### ✅ 优势

1. **职责清晰**
   - 每个文件专注单一功能
   - 符合SOLID原则
   - 易于理解和维护

2. **易于测试**
   - 小函数易于单元测试
   - Mock依赖更简单
   - 测试覆盖率更高

3. **团队协作**
   - 多人并行开发不冲突
   - Git merge冲突更少
   - Code review更聚焦

4. **可扩展性**
   - 添加新功能只需新增文件
   - 不影响现有代码
   - 遵循开闭原则

5. **功能完整**
   - ✅ 包含Legacy功能
   - ✅ 仓库列表和克隆

#### ❌ 劣势

1. **导航复杂**
   - 需要在多个文件间跳转
   - IDE索引压力大
   - 查找实现需要时间

2. **维护成本**
   - 修改功能可能涉及多个文件
   - 导入路径较长
   - 文件管理成本高

3. **打包体积**
   - 更多的文件意味着更多的模块元数据
   - Tree-shaking可能不够彻底

---

### `github/` 扁平架构优势

#### ✅ 优势

1. **快速导航**
   - 相关代码集中在单文件
   - 减少文件切换
   - 提升开发效率

2. **部署简单**
   - 更少的文件依赖
   - 打包更快
   - 减少模块加载开销

3. **代码连贯性**
   - 一次性浏览完整实现
   - 更容易理解业务逻辑
   - 减少上下文切换

4. **IDE友好**
   - 单文件搜索更快
   - 自动补全更准确
   - 减少符号解析开销

#### ❌ 劣势

1. **文件过大**
   - `code.ts` 644行
   - `account.ts` 604行
   - 超出单屏显示范围

2. **职责混合**
   - `account.ts` 包含models+stores+routes+settings
   - 违反单一职责原则
   - 难以进行局部重构

3. **测试困难**
   - 大文件导致测试文件也变大
   - Mock依赖更复杂
   - 单元测试粒度粗

4. **团队协作冲突**
   - 多人同时修改同一文件易冲突
   - Code review范围过大
   - Git diff难以阅读

5. **功能缺失**
   - ❌ 缺少 `listGitHubRepositories`
   - ❌ 缺少 `cloneGitHubRepository`
   - 影响功能完整性

   - `g/` = 传统分层架构 (46+个文件)
   - `github/` = 现代扁平架构 (9个文件)

2. **功能完整性：**
   - `g/` = 40个功能 (100%)
   - `github/` = 38个功能 (95%)
   - **缺失2个Legacy功能**

3. **代码密度：**
   - `g/` = 分散实现 (每文件50-200行)
   - `github/` = 集中实现 (每文件最高644行)

4. **维护性：**
   - `g/` = 易于测试和扩展，导航复杂
   - `github/` = 快速查看，大文件难维护

### 推荐方案

#### 📌 方案A：保留 `g/` 目录（推荐用于企业级项目）

**适用场景：**
- 团队规模 > 3人
- 需要频繁迭代和扩展
- 重视代码质量和可测试性
- 需要完整功能（包括Legacy）

**优势：**
- ✅ 更好的可维护性
- ✅ 更清晰的职责分离
- ✅ 更容易的单元测试
- ✅ 功能100%完整

---

#### 📌 方案B：使用 `github/` 目录（推荐用于小型项目）

**适用场景：**
- 个人项目或小团队
- 功能相对稳定，不常扩展
- 不需要Legacy功能
- 追求快速开发和部署

**优势：**
- ✅ 更快的导航速度
- ✅ 更简洁的文件结构
- ✅ 更少的文件管理开销

**需要：**
- ⚠️ 添加Legacy功能支持
- ⚠️ 考虑拆分超大文件（>500行）

---

#### 📌 方案C：混合架构（平衡方案）

**建议：**
- 保留 `g/` 的分层结构
- 合并过小的文件（<50行）
- 将超大文件拆分（>400行）
- 标准单文件大小：100-300行

**实施步骤：**
1. 保留models、stores、api独立文件
2. 每个router模块一个文件（200-300行）
3. 添加Legacy支持到github/
4. 最终形成12-15个文件的中等规模结构
